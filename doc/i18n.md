# UI i18n Notes

This document explains how the current frontend i18n setup works, what must be updated when adding UI text, and the common pitfalls that caused untranslated labels during recent work.

## Current structure

There are currently two i18n files in use:

- `ui/src/lib/i18n.ts`
- `ui/public/scripts/lib/i18n.js`

They are both important.

### `ui/src/lib/i18n.ts`

This is the source-side i18n module used by the TypeScript code in `ui/src`.

It provides:

- `t(key, params?)`
- `applyI18n()`
- `injectLangToggle()`
- localStorage persistence via `polar_lang`

### `ui/public/scripts/lib/i18n.js`

This is the browser-served runtime i18n file actually loaded by pages under `ui/public/*.html`.

If a label exists only in `ui/src/lib/i18n.ts` but not here, the browser may show raw keys like:

- `dashboard.loadMore`
- `login.passkeyHint`
- `editor.heading`

That was the main reason language switching looked partially broken.

## Default language

The current default language is English.

Behavior:

- if `localStorage["polar_lang"]` is unset, fallback is `en`
- language toggle writes `polar_lang`
- page reload should keep the selected language

## How page translation works

Most HTML pages use:

- `data-i18n`
- `data-i18n-placeholder`
- `data-i18n-title`

Those attributes are applied at runtime by:

- `applyI18n()` inside `ui/public/scripts/lib/site.js`

Important:

- `ui/public/scripts/lib/site.js` must import and call `applyI18n()`
- `ui/public/scripts/lib/site.js` must also import `t()` if it renders translated strings itself

Current runtime flow:

1. page loads `.../scripts/*.js`
2. that script calls `hydrateSiteBrand()`
3. `hydrateSiteBrand()` calls `applyI18n()` and `injectLangToggle()`
4. HTML labels with `data-i18n*` are updated

If a page has `data-i18n` markup but its script never reaches `hydrateSiteBrand()`, labels will not switch.

## When adding new UI text

For any new user-facing label, check all of these:

1. Add the key to `ui/src/lib/i18n.ts`
2. Add the same key to `ui/public/scripts/lib/i18n.js`
3. Prefer `data-i18n` in HTML for static text
4. Use `t("...")` for text generated in JS
5. Avoid hard-coded Chinese or English in runtime JS

## Pages already using i18n markup

Examples:

- `ui/public/dashboard.html`

- `ui/public/register.html`
- `ui/public/editor.html`

For these pages, missing runtime keys in `ui/public/scripts/lib/i18n.js` will break visible translation even if `ui/src/lib/i18n.ts` is correct.

## Common pitfalls

### 1. Updating only `ui/src/lib/i18n.ts`

This is not enough for the current public runtime.

Symptom:

- page shows raw key names instead of translated text

Examples:

- `dashboard.loadMore`
- `login.passkeyHint`
- `editor.heading`

### 2. Hard-coded strings in HTML

Example problem:

- section badge text written directly as `LLM Config`
- section title written directly in Chinese without `data-i18n`

Fix:

- move to `data-i18n`
- add matching keys in both i18n files

### 3. Hard-coded strings in JS

Example problem:

- `element.textContent = "网络错误，请重试"`
- `element.textContent = "站点管理"`

Fix:

- replace with `t("...")`

### 4. Merge conflicts mixing `setStatusMessage(...)` and `textContent`

Preferred pattern:

- use `setStatusMessage(...)` where the UI already supports success/error styling
- use `t("...")` for all user-visible copy

Example preference:

- good: `setStatusMessage(passkeyStatus, t("dashboard.passkeyFailed"), "error")`
- avoid: `passkeyStatus.textContent = "Passkey 绑定失败"`

## Recommended workflow for continued i18n work

When you touch a page:

1. Search the page HTML for `data-i18n`
2. Search its script for hard-coded Chinese or English
3. Search `ui/public/scripts/lib/i18n.js` for the keys used by that page
4. Search `ui/src/lib/i18n.ts` for the same keys
5. Make both files consistent

Helpful commands:

```bash
rg -n 'data-i18n|data-i18n-placeholder|data-i18n-title' ui/public
rg -n '[\p{Han}]' ui/public/scripts ui/public/*.html
rg -n 'editor\.|dashboard\.|login\.|register\.|posts\.|post\.|markdowns\.|markdown\.' ui/public/scripts/lib/i18n.js ui/src/lib/i18n.ts
```

## Future cleanup suggestion

The current setup duplicates translation data across:

- `ui/src/lib/i18n.ts`
- `ui/public/scripts/lib/i18n.js`

Longer term, this should be unified so public runtime assets are generated from one source of truth instead of being maintained by hand in two places.
